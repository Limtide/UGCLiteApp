# UGCLite-图文客户端

## 业务背景

图文业务的目标是建设一个能满足好普通人的内容创作与交流的 UGC 社区，解决好用户的经验决策、圈层交流类需求。UGC 强调内容的真实多元、作者的第一人称表达，相较于 PGC 类内容，更加多元，由普通人分享创作，因为每个人都有自己的生活方式，通过抖音双列及/可颂产品以内容与兴趣做好社区的人与人连接。

产品形态上，以双列图文视频混排，内容体裁以图文为主。双列区别于无限上下滑方式，双列的屏效比更高，一屏展示多个作品更能满足好用户的内容筛选与消费决策。图文体裁具有信息密度高、可控感强的优势，帮助用户更快速的进行内容消费与筛选，并且图文体裁内容创作成本低，利于普通用户的低门槛创作。

## 一、实现方案

1. 根据功能划分了4个模块进行开发。
   1. 主界面模块。
   2. 详情页模块。
   3. 主界面中间内容。
   4. 登录模块。
2. 开发的顺序是先构建界面，在Activity或者Fragment为界面绑定后，再进行业务逻辑的开发。业务逻辑主要分为1.处理交互，2.与网络交互，3.与本地数据库交互。
3. 根据MVVM架构思想，进行分层。![Android App MVVM](https://raw.githubusercontent.com/Limtide/mock_post_data/refs/heads/main/db/Android%20MVVM.svg)
   1. View层：
      - 主界面：使用Activity，作为主容器，管理底部导航和 Fragment切换。
      - 详情页模块：根据不同Activity负责不同模块的思想，使用Activity。
      - 登录模块：登录作为一个完整的业务流程，可以单独使用Activity。
      - 主界面中间内容：作为主界面的子内容，与主界面生命周期高度绑定，所以使用Fragment。
   2. ViewModel层：
      - 在实现过程中，只实现了LoginViewModel，通过LiveData双向绑定了LoginActivity，能够直接检查内容。
      - 实际应该再将HomeViewModel，PostDetailViewModel补充。
   3. Model层：
      - FeedResponse：先针对API返回的内容进行接收，包括状态码和post列表。
      - Post：然后是Post，作为UGCLite的最多的数据，通过API获取，观察返回的数据后，选择只让Post对应post_list中的内容。
      - 为了实现登录，又设计了User实体，为了简化开发，直接用username作为主键，本来想做一个记住代码自动登录的，所以设计了最后登录时间，超过多少天之后不在记住密码，但是最后没有完成
   4. Repository层
      - 实际在开发过程中，因为没有后端可以调用数据库查询用户，所以一开始是想着将用户存储在本地，直接可以调用，但是后面在小红书app登录的时候发现，用户信息还是要存储在远程上的。这种既有远程又有本地的操作起来比较麻烦，而且也不好让上层掉用，所以添加了Repository层，实现了UserRepository，想着能够在远程验证用户身份后，保存用户的令牌或者登录状态能够方便一点。
   5. 网络层：
      - 设立一个网络层，主要负责向服务器请求数据和发送更改，其他模块只需要调用他就行，不用将这些内容都塞到其他代码里面。
   6. 工具包：
      - 开发过程中，有些工具可以在其他项目里面复用，也有可能需要在其他多个代码中复用，所以我将这些代码提取出来。独立成类让其他调用。
        - 为了保存一些用户的信息，用户的偏好设置，比如设置了用户名，静音状态，自动播放等等，设置了专门一个管理类。
        - 点赞管理和关注管理，可以在多个界面应用，比如在主页面和详情页有点赞，详情页和其他用户主页有关注，需要多次使用，未来也可以作为一个单独的线程来向服务器发送点赞和关注状态。
        - 登录密码存储使用MD5加密，在存储和未来在网络发送验证的时候，不会泄漏明文。
        - 后面再完成进阶功能的时候，遇到了音频播放，详情页的内容已经被我写的很多了，所以我新设置了一个工具来缓存MusicFileUtils。
        - 解决了音频缓存后得不到清理的问题，我在边刷，边观察系统提供的app info，cache占用的地方，发现了刷的过程中，app会无线增长，在退出后有所缓解，推测是缓存的Post没有进行显示，于是限制了Post的存储列表，以及一并修复了视频缓存图清理的问题。

#### API接口

##### 数据管理

数据存储在https://github.com/Limtide/mock_post_data。使用Vercel作为简单的API接口托管。

##### 接口信息

- **接口路径：** `https://yeduguzhou.com/api`
- **请求方法：** `GET`
- **接口描述：** 获取社交媒体内容推送，支持图片和视频内容

##### 请求参数

| 参数名       | 类型    | 必填 | 默认值 | 说明                                  |
| ------------ | ------- | ---- | ------ | ------------------------------------- |
| count        | integer | 否   | 10     | 返回内容数量，建议范围1-50，最大为200 |
| accept_video | boolean | 否   | false  | 是否包含视频内容                      |

##### 请求示例

```http
GET /api?count=20&accept_video=true HTTP/1.1
Host: https://yeduguzhou.com
Content-Type: application/json
```

## 三、个人心得

1. 使用TableLayout实现顶部导航栏后，发现导航栏不能左右滑动，从[掌握Android TableLayout滑动技巧，轻松实现动态布局设计](https://www.oryoy.com/news/zhang-wo-android-tablelayout-hua-dong-ji-qiao-qing-song-shi-xian-dong-tai-bu-ju-she-ji.html)中找到了解决方法，使用HorizontalScrollView将view包裹后，可以实现滑动。
2. RelativeLayout与FrameLayout的区别：RelativeLayout通过子控件之间的相对位置关系和子控件与父布局的相对位置关系来定位，FrameLayout其内部的控件默认会被叠放至布局的左上角，并且后添加的控件会覆盖先添加的控件。
3. 试图在启动页选择联网和断网模式，利用Intent实现向MainActivity跳转并传递信息。但出现了错误：
   1. 我设置了AlertDialog选择，但并没有观察到而是直接跳转。
      1. 解决方法：将跳转方法放入选项中。
   2. 传递过程中出现android.view.WindowLeaked，因为SplashActivity已经onDestroy，但我在AlertDialog中设置的Toast仍然存在。
      1. 解决方法：将跳转方法放入选项中，没在日志中看到问题，但是我并没有想明白这是为什么？
4. 使用教程[实现Android下拉刷新](https://blog.csdn.net/gitblog_01195/article/details/155028406)实现了下拉刷新，但是目前下拉后一直刷新，还没搞明白为什么。
   1. 解决方法：更正绑定的监听器，监听器写的有问题。
5. 在[UI命名](https://zhuanlan.zhihu.com/p/34336714)中学习命名格式：控件\类别\功能\状态@倍率.png
6. 使用Fragment取代ProfileActivity，实现保留底部导航栏切换主页面内容。
7. 防止ViewModel导致的内存泄漏：ViewModel 绝不能持有 View、Activity、Fragment 的引用，也不能持有包含 Activity Context 的对象 (比如 Drawable)。
   1. 屏幕旋转，旧 Activity 销毁。
   2. 但是 ViewModel 因为机制原因，活了下来。
   3. 这个活着的ViewModel持有抓着旧Activity的引用
   4. 结果：旧 Activity，GC无法回收，内存泄漏。
8. ViewModel的生命周期是绑定在Activity的逻辑生命周期上的，而不是物理生命周期。
9. 观察与监听，为什么能进行：
   1. 系统预先定义好了一个接口，如 OnClickListener。
   2. 代码中实现实现逻辑，通过调用了button.setOnClickListener(监听器对象)，实现绑定。
10. Fragment生命周期：
    1. 使用replace，会直接杀死了Fragment，结束并回收Fragment，触发onDestroyView()，无法储存状态。每次替换后容器内只有一个 Fragment存在。
    2. 使用add，原有 Fragment 保持 onResume() 状态。多个 Fragment 会叠加显示。
11. 在onSuccess里直接setText()报错，提示不能在非UI线程操作View。网络回调是在子线程的，得切回主线程或者用Handler/LiveData来更新UI。通过Room的LiveData监听数据库变化来刷新UI，比较好用。
12. 定义实体类User时，Room一直报错。Room需要一个默认的无参构造函数。保留默认构造函数，自己写的带参数的业务构造函数加上@Ignore注解。
13. ReentrantLock与synchronized不同，ReentrantLock显示调用锁，synchronized隐式获取锁。ReentrantLock比synchronized更灵活。其实用起来ReentrantLock比synchronized复杂点，得手动管理锁，synchronized能自动释放。
14. 使用Vercel作为简单的API接口托管，使用github作为存储平台。但现在Vercel默认 IP 被 DNS 污染或阻断，所以利用 Cloudflare 作为中间反向代理，连接阿里云管理的域名与 Vercel 服务器。使用GitHub进行静态资源存储，未来计划用jsDelivr CDN作加速。
